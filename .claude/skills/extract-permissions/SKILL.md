# Extract Permissions from Local Repositories

## Goal

Scan `.claude/settings.local.json` files across all local repositories managed
by ghq, analyze their permission entries, and produce a consolidated set of
universal permissions using wildcard patterns for the home-manager claude-code
module at `modules/home/coding-agents/claude-code/default.nix`.

## Workflow

### Step 1: Discover settings files

```bash
find "$(ghq root)" -name "settings.local.json" -path "*/.claude/*" 2>/dev/null
```

### Step 2: Run the analysis script

```bash
nix run nixpkgs#python3 -- ~/.claude/skills/extract-permissions/scripts/analyze.py
```

The script will:
1. Find all `.claude/settings.local.json` under `$(ghq root)`
2. Parse each file and extract `permissions.allow`, `permissions.deny`, `permissions.ask`
3. Normalize and categorize each entry (Bash, WebFetch, Read, Edit, MCP, etc.)
4. Count occurrences across repos
5. Identify candidates for promotion to user-level settings

### Step 3: Review and present results

Present results grouped by category:

| Category | Description | Wildcard Strategy |
|----------|-------------|-------------------|
| **Read-only shell** | `ls`, `cat`, `tree`, `head`, `find`, `grep`, `rg` | `Bash(cmd *)` per command |
| **Nix commands** | `nix build`, `nix log`, `nix eval`, `nix flake check` | `Bash(nix *)` or per-subcommand |
| **Git read** | `git log`, `git show`, `git branch`, `git diff` | `Bash(git log *)` etc. |
| **GitHub CLI** | `gh pr`, `gh issue`, `gh run`, `gh api` | `Bash(gh *)` with subcommand wildcards |
| **WebFetch domains** | Trusted documentation sites | `WebFetch(domain:example.com)` |
| **MCP tools** | Server-specific tool permissions | `mcp__server__*` or individual |
| **Build/test** | `cargo`, `pnpm`, `npm`, `deno`, `just` | Per-ecosystem wildcards |

### Step 4: Generate recommendation

Output a Nix list suitable for adding to
`modules/home/coding-agents/claude-code/default.nix` under
`settings.permissions.allow`. Include:

- Entries appearing in 3+ repos (strong candidates)
- Entries appearing in 2 repos with read-only semantics (good candidates)
- Read-only commands that are safe universally
- Trusted documentation domains

Exclude:
- Project-specific paths (absolute `Read(//home/...)` entries)
- Project-specific build commands with hardcoded paths
- One-off commands that were clearly ad-hoc approvals

### Step 5: Diff against current settings

Compare the recommendation against current `~/.claude/settings.json`
(generated by home-manager) to show:
- Already covered (no action needed)
- New additions (recommend adding to the Nix module)
- Potentially removable from local settings (already in user-level)

## Output Format

```
## Permission Analysis Report

### Summary
- Scanned: N repositories
- Total unique permissions: N
- Candidates for user-level: N

### Recommended additions to claude-code module
(Nix list of permission strings)

### Already in user-level settings
(list of permissions that are already covered)

### Remaining project-specific permissions
(permissions that should stay in local settings)
```

## Wildcard Pattern Reference

Permission rules follow the format `Tool` or `Tool(specifier)`.
Bash rules support glob patterns with `*` at any position.

### Bash wildcards

| Pattern | Matches | Does NOT match |
|---------|---------|----------------|
| `Bash(npm run *)` | `npm run build`, `npm run test` | `npm install` |
| `Bash(git * main)` | `git checkout main`, `git merge main` | `git push origin dev` |
| `Bash(* --version)` | `node --version`, `cargo --version` | `node --help` |
| `Bash(ls *)` | `ls -la`, `ls /tmp` | `lsof` (word boundary enforced by space before `*`) |
| `Bash(ls*)` | `ls -la`, `lsof` | — (no word boundary) |

Key rules:
- The space before `*` matters: `Bash(ls *)` enforces a word boundary, `Bash(ls*)` does not
- The legacy `:*` suffix syntax is equivalent to ` *` but is deprecated — prefer `Bash(cmd *)` over `Bash(cmd:*)`
- Shell operators like `&&` are handled: `Bash(safe-cmd *)` won't match `safe-cmd && other-cmd`
- `Bash(*)` is equivalent to `Bash` and matches all commands

### Other tool patterns

| Pattern | Effect |
|---------|--------|
| `WebFetch(domain:example.com)` | Matches fetch requests to example.com |
| `Read(/src/**)` | Reads from `<project>/src/` (`*` = single dir, `**` = recursive) |
| `Edit(~/.config/**)` | Edits under home config directory |
| `mcp__server__*` | All tools from an MCP server |
| `mcp__server__tool_name` | Specific MCP tool |

### Precedence

Rules are evaluated in order: **deny → ask → allow**. The first matching rule wins.

Reference: https://code.claude.com/docs/en/permissions#wildcard-patterns

## Important Notes

- Never auto-modify the home-manager module — only recommend changes
- Prefer specific wildcards over overly broad ones (e.g., `Bash(nix build *)` over `Bash(nix *)`)
- Prefer `Bash(cmd *)` (space-wildcard) over deprecated `Bash(cmd:*)` (colon-wildcard)
- Read-only and trusted-source WebFetch domains are safe to promote
- MCP tool permissions depend on which servers are configured per-project
